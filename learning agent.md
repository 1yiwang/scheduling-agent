# Learning Agent · Agent vs Workflow，以及产品如何越用越聪明

> 本文记录核心方法论讨论：这个 scheduling agent 到底是什么、如何构建、
> feedback loop 如何设计与新增、以及为什么"数据分析"才是让产品进化的引擎。
> 配套见 `BRAINSTORM.md`（具体功能决策）。

---

## 一、Agent 还是 Workflow？——它两者都是

底层是 **workflow**，对外表现是 **agent**。不要二选一。

> **这个产品 = 一个持续运行的时间决策系统。**
> Workflow 是它的执行骨架，Agent 是它对外表现出来的智能行为。

### Workflow 层（它怎么做事）
固定的五步循环：
1. **感知**：发现一个时间机会/问题（如：明早有段火车通勤、有人要约会、今天太挤）。
2. **拉上下文**：日历、地点、通勤方式、待办、欠回复、会议重要性。
3. **确定性判断**：这段通勤能不能用？放电话还是视频？任务时长够不够？
4. **给候选方案**：如 3 个 tab——给 Alex 打电话 / 读投资人 memo / 准备 board notes。
5. **用户确认**：单人任务直接排进去；涉及别人则生成消息草稿。
6. **学习结果**：记下用户选了什么、拒了什么、哪类通勤常被用来开会。

### Agent 层（它为什么像"助理"）
只有 workflow 是死的（用户点按钮才动）。Agent 多了三件事：
- **主动性**：自己发现"这段通勤可用"，不等你问。
- **上下文感**：知道火车/公交/飞机/开车的差异。
- **记忆**：逐渐知道你喜欢把哪些事放进通勤。

### 产品层（用户感受到什么）
用户不该感到"我在跑 workflow"，而是：
> "它懂我的时间，并在关键时刻给我一个可接受的安排。"

## 二、Agent 如何构建——故意把 LLM 踢出大脑

一句话：**LLM 听话和说话，引擎做决定，框架管节奏和授权。**

五器官循环，只有两处碰 LLM：

| 器官 | 干什么 | 谁来做 |
|---|---|---|
| 感知 | 读日历/算通勤窗口；解析乱输入 | 确定性 / LLM（耳） |
| **决策** | 候选池→模态门控→适配→排序→Top3 | **纯确定性 · 大脑** |
| 表达 | 写人话 / 写给对方的草稿 | LLM（嘴） |
| 行动 | 排进通勤、发消息 | 确定性 + autonomy 滑块 |
| 记忆 | 记接受/拒绝、更新偏好 | 确定性 |

- **大脑 = 确定性引擎 + 记忆**（你的 IP，别人 fork 不走）。
- **LLM = 边缘**（耳朵 + 嘴），可换、便宜、小模型够，绝不让它自由决定排什么。
- **agent = 编排循环 + 授权策略**，不替你做调度决定。

### 与 Hermes / OpenClaw 的本质区别
| | Hermes / OpenClaw | 本产品 |
|---|---|---|
| LLM 角色 | **大脑**——自己决定干啥 | **边缘**——只解析 + 表达 |
| 框架 | 厚重、通用、自主 | 薄、专用、确定性兜底 |
| 结果 | 强但不可靠、要调教 | 可靠、零描述、敢放权 |

Hermes 把 LLM 当大脑 → "什么都能干但算不准、要你检查"；
本产品用代码当大脑、LLM 在边缘 → **敢让它直接排你的时间还不出错**。这是能赢它的点。

## 三、Feedback Loop 如何设计与新增（最关键的认知）

### 先拆开"agent"模糊掉的三层
| 层 | 是什么 | 谁来定 | 该不该固定 |
|---|---|---|---|
| **流程骨架** | 感知→决策→行动的步骤 | **你写死** | ✅ 该固定，可靠性来自它固定 |
| **骨架里的参数** | 偏好权重、每条路线/模态的默认值 | **数据学** | ❌ 结构固定，**值在动** |
| **"该学什么"本身** | 要不要加一个新反馈维度 | **你 + 数据**（元过程） | ❌ 但不是 agent 在生产里自己发明 |

"feedback loop 固定吗"这个担心，混的是第 2 层和第 3 层。

### 三层各自的精确答案
- **第 1 层（流程）**：是的，就是"我们梳理清楚后写进去"，而且**这是对的**。没法从零学出一个靠谱的调度器，领域常识必须人编码。这叫工程，不丢人。
- **第 2 层（参数）**：这里有真学习，但"结构固定、值在变"。"火车上能开视频"这条规则的结构是你写的；"这位用户在通勤上接受视频会的概率"是数据喂出来的。
- **第 3 层（新 loop）**：短期加不了、也**故意不加**——早期你根本不知道哪个反馈重要，先把最明显的 loop（接受/拒绝 → 更新偏好）做扎实。

### 新 loop 怎么加——靠"元循环"，不靠 agent 自我发明
> 记日志 → 你（+ LLM 当分析师）看出新模式 → 决定加一个新信号 → 上线。
>
> **agent 不进化，公司进化；公司进化了，agent 才进化。**

### LLM 在哪真正帮你突破"固定"
把 LLM 当**离线分析师**，不当在线决策者：
> 让它读积累的接受/拒绝日志，**帮你发现该加哪条新规则 / 新反馈**——
> 例如"用户在公交上一直拒视频会，虽然我们允许" → 提示你加一条新门控。

这样：在线决策仍确定性（保住可靠），但"该学什么"靠 LLM + 你一起挖（突破"只能学预定义"的天花板）。**LLM 不在产品里乱决策，而在你的迭代循环里帮你变聪明。**

## 四、比架构更重要的：数据飞轮

逻辑可以被抄，**积累的偏好数据 + 你迭代的速度**抄不走。

1. **从第一天就埋点**：记录每一次"建议 + 用户反应"（接受 / 拒绝 / 改了什么 / 无视）。
   没有日志，永远没法变聪明——这比任何聪明算法都重要。
2. **赢家不是逻辑最巧的，是信号抓得对、迭代最快的。**
3. **学习的艺术是"选少数关键反馈"，不是建无限 loop。** 克制——只在数据告诉你"这个维度真的重要"时才加。

## 五、数据分析层（未来）——告诉你"哪里该改进 / 加反馈"

埋点只是采集；要让产品自己告诉你下一步改哪，需要一个分析层。

### 该记录的信号（埋点 schema 雏形）
每次 agent 给出建议时记一条事件：
- **建议上下文**：通勤段（路线/模态/时长）、候选池来源、Top3 是什么、排序依据。
- **用户反应**：接受 / 拒绝 / 选了第几个 / 改了什么 / 直接无视 / 多久后反应。
- **执行结果**（若可得）：排进去的事后来发生了吗 / 被改期 / 被删。
- **授权状态**：草稿手动发 vs 授权自动发（autonomy 滑块位置）。

### 分析层要回答的问题
- 哪类通勤窗口的建议**接受率最高/最低**？（→ 哪些门控规则不准）
- 用户**总选第几个** tab？（→ 排序逻辑是否需要调权重）
- 哪些候选池来源贡献了被接受的建议？（→ 哪个池值得加强/砍掉）
- 用户**反复手动改**的是什么？（→ 隐藏的偏好，可能是一条该新增的规则）
- 哪些场景用户从手动授权升级到自动授权？（→ 信任在哪建立，autonomy 可在哪放权）

### 闭环
> 埋点采集 → 分析层暴露"接受率低 / 反复被改 / 总被无视"的点
> → 人 + LLM 判断该调参数还是加新 loop → 上线 → 再采集。

这就是让产品**越用越聪明**的真正引擎——不是某个聪明模型，而是这个**可观测、可迭代的闭环**。

---

## 六、一句话收尾
> **流程你写死（对），参数让数据调（真学习），
> 新反馈靠"记日志 → 人 + LLM 分析 → 上线"的元循环加（不是 agent 自己发明）。
> 产品真正的护城河不是那套逻辑，是日志飞轮 + 数据分析 + 你的迭代速度。**

### 对下一步的直接影响
Prototype 从第一版就要**埋点记录每次建议和反应**，哪怕学习逻辑还很简单——
否则数据飞轮永远转不起来。
