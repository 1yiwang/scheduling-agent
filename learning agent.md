# Learning Agent · Agent vs Workflow，以及产品如何越用越聪明

> 本文记录核心方法论讨论：这个 scheduling agent 到底是什么、如何构建、
> feedback loop 如何设计与新增、以及为什么"数据分析"才是让产品进化的引擎。
> 配套见 `BRAINSTORM.md`（具体功能决策）。

---

## 一、Agent 还是 Workflow？——它两者都是

底层是 **workflow**，对外表现是 **agent**。不要二选一。

> **这个产品 = 一个持续运行的时间决策系统。**
> Workflow 是它的执行骨架，Agent 是它对外表现出来的智能行为。

### Workflow 层（它怎么做事）
固定的五步循环：
1. **感知**：发现一个时间机会/问题（如：明早有段火车通勤、有人要约会、今天太挤）。
2. **拉上下文**：日历、地点、通勤方式、待办、欠回复、会议重要性。
3. **确定性判断**：这段通勤能不能用？放电话还是视频？任务时长够不够？
4. **给候选方案**：如 3 个 tab——给 Alex 打电话 / 读投资人 memo / 准备 board notes。
5. **用户确认**：单人任务直接排进去；涉及别人则生成消息草稿。
6. **学习结果**：记下用户选了什么、拒了什么、哪类通勤常被用来开会。

### Agent 层（它为什么像"助理"）
只有 workflow 是死的（用户点按钮才动）。Agent 多了三件事：
- **主动性**：自己发现"这段通勤可用"，不等你问。
- **上下文感**：知道火车/公交/飞机/开车的差异。
- **记忆**：逐渐知道你喜欢把哪些事放进通勤。

### 产品层（用户感受到什么）
用户不该感到"我在跑 workflow"，而是：
> "它懂我的时间，并在关键时刻给我一个可接受的安排。"

## 二、Agent 如何构建——故意把 LLM 踢出大脑

一句话：**LLM 听话和说话，引擎做决定，框架管节奏和授权。**

五器官循环，只有两处碰 LLM：

| 器官 | 干什么 | 谁来做 |
|---|---|---|
| 感知 | 读日历/算通勤窗口；解析乱输入 | 确定性 / LLM（耳） |
| **决策** | 候选池→模态门控→适配→排序→Top3 | **纯确定性 · 大脑** |
| 表达 | 写人话 / 写给对方的草稿 | LLM（嘴） |
| 行动 | 排进通勤、发消息 | 确定性 + autonomy 滑块 |
| 记忆 | 记接受/拒绝、更新偏好 | 确定性 |

- **大脑 = 确定性引擎 + 记忆**（你的 IP，别人 fork 不走）。
- **LLM = 边缘**（耳朵 + 嘴），可换、便宜、小模型够，绝不让它自由决定排什么。
- **agent = 编排循环 + 授权策略**，不替你做调度决定。

### 与 Hermes / OpenClaw 的本质区别
| | Hermes / OpenClaw | 本产品 |
|---|---|---|
| LLM 角色 | **大脑**——自己决定干啥 | **边缘**——只解析 + 表达 |
| 框架 | 厚重、通用、自主 | 薄、专用、确定性兜底 |
| 结果 | 强但不可靠、要调教 | 可靠、零描述、敢放权 |

Hermes 把 LLM 当大脑 → "什么都能干但算不准、要你检查"；
本产品用代码当大脑、LLM 在边缘 → **敢让它直接排你的时间还不出错**。这是能赢它的点。

## 三、三层自我改进架构（Agent 如何越用越聪明）

核心目标：**Agent 能在运行时自我提高——确定的自动执行，不确定的问人。**

不同维度的变化不需要同一个机制。按"发现难度"分三层，每层有不同的自主权和通知策略。

### 先拆清三层
| 层 | 做什么 | 谁来定 | 自主权 | 通知用户？ |
|---|---|---|---|---|
| **Tier 1 · 参数自调** | 已知维度上变权重 | **数据学**（全自动） | ✅ 自动执行 | ✅ 微提示告知 |
| **Tier 2 · 模式发现** | 发现现有规则没覆盖的新行为模式 | **统计分析 + 置信度门控** | ⚠️ 高置信自动 / 低置信建议 | ✅ 高置信通知 / 低置信询问 |
| **Tier 3 · 结构性学习** | 发现需要新种类的反馈维度 | **人 + LLM** 离线分析 | ❌ 人拍板 | Agent 主动汇报，人决策 |

---

### Tier 1 · 参数持续自调（全自动）

**能自动调的参数：**

| 参数 | 信号来源 | 学习方式 |
|------|---------|---------|
| 路线→workability（火车视频/公交电话） | 接受 N 次同类建议 | 统计显著性达标 → 自动更新默认值 |
| 时段→偏好权重（早上不排会） | 拒绝 N 次该时段建议 | 连续拒绝 → 自动降权 |
| 人→预估时长 | 手动修改时长 | 观测实际时长 → 调预估 |
| 事件类型→默认 buffer | 手动改 buffer | 多次同一类型被改 → 调默认值 |
| 候选池→排序权重 | 选第 N 个/拒绝全部 | 选+1/拒-1 → 但不止于此 |

**升级版学习（不只是选+1/拒-1）：**

```
旧：选视频会 → videoCallWeight +1
新：7 次同一路线火车上选了视频会 → 计算统计 →
    confidence = f(n=7, consistency=100%, recency=30d) = 96% →
    自动将该路线 workability.video 从 0.5 提到 0.9 →
    通知："这条路线我标记为视频可用 · 最近 7/7 次你都接受了"
```

**Tier 1 的自主权：** 自动执行，不等人拍板。但每次调整都附一条微提示——用户知道 agent 在变，也能手动撤回。

---

### Tier 2 · 新模式发现 + 置信度门控（核心新增 ⭐）

Agent 定期扫日志，用统计方法发现**现有规则没覆盖的行为模式**。

**关键创新：agent 不等人来加新维度——它自己发现、自己提议、高置信时自己执行。**

**信号来源（丰富且结构化，不需要 LLM）：**

| 信号 | 含义 | 可能触发的新规则 |
|------|------|---------------|
| ✅ 选了第 N 个 | 排序偏好 | 调整该类型/人的权重 |
| ❌ 拒绝全部建议 | 候选池或门控有问题 | 检查该窗口的模态门控是否过严 |
| ✏️ 手动改了时长 | 时长估算不准 | 该人/类型的默认时长需调整 |
| ✏️ 手动加了 buffer | buffer 规则不够 | 该事件类型/地点的 buffer 默认值需调 |
| 👻 直接无视 | 触发时机/优先级不对 | 该时段的建议触发太频繁或类型不匹配 |
| 🔁 改后自己又改回来 | 用户不喜欢 agent 的改动 | 该维度不应被自动调整 |

**置信度门控逻辑：**

| 置信度 | 动作 | 用户体验 |
|-------|------|---------|
| ≥ 90% | **自动应用** + 通知 | "我发现你常拒绝午饭时间的会——已自动保护 12:00-13:00" |
| 60–90% | **主动建议** + 等你确认 | "你最近 8/10 次在公交上接受了音频会——要将公交标记为音频可用吗？" |
| < 60% | **静默积累数据** | 不打扰你，后台继续采集 |

**具体例子：**

```
发现：过去 2 周，10 次"午饭时间 12-13"的建议你拒了 9 次
现有规则：没有"午饭保护"这个维度 →
统计：n=10, 拒绝率=90% → confidence = 94% →
动作：≥90% → 自动新建规则："12:00-13:00 不排会" →
通知："午饭时间我自动保护了 · 最近 10 次你拒了 9 次"

发现：公交上你接受了 2/3 音频会，但拒了 5/5 视频
现有规则：公交 workability = 音频+电话可，视频不可 →
统计：模式已被现有规则覆盖 →
动作：无新规则，仅加固现有规则的置信度
```

**Tier 2 的自主权：** 高置信自动执行并通知，低置信主动询问——agent 不需要人来"发现"，只在高不确定性的边缘才找人确认。**这是从"人驱动进化"到"agent 驱动进化"的关键跃迁。**

---

### Tier 3 · 结构性学习（人参与）

**"需要新维度吗？我帮你发现了——但需要你来设计结构。"**

这是"新种类反馈"——不属于任何现有维度。Agent 发现异常但不会自己处理：

```
Agent（每周/每月报告）："我追踪了 30 天数据，发现 3 个我无法自动优化的模式：
  1. 线下会的 buffer 你常手动加长 · 不属于标准 buffer 规则 ·
     建议：为线下会新增独立 buffer 参数
  2. 和 UZH 相关的人见面，你倾向约在大学附近 · 我没有'人→偏好地点'维度 ·
     建议：新增 person.preferredLocation 参数
  3. ..."

你 → 拍板："做 #1 和 #2"
Agent → 创建新参数结构 → 自动归入 Tier 1/2 → 后续自主优化
```

**LLM 的角色（离线分析师，非在线决策者）：**
> 读积累的日志 → 帮你发现"我漏了什么新维度"→ 生成报告 →
> 你决定加不加 → 上线后归入 Tier 1/2 自动运转。

LLM 不跑在线决策（保住可靠），但在迭代循环里加速你的判断。

---

### 三层协同示意

```
                    ┌─────────────────────────────┐
                    │      Tier 3 · 结构性学习      │
                    │   人 + LLM 离线 · 月度/季度    │
                    │   发现需要新维度 → 人拍板      │
                    └──────────┬──────────────────┘
                               │ 上线新参数结构
                               ▼
               ┌───────────────────────────────┐
               │    Tier 2 · 模式发现            │
               │   统计分析 · 每周自动扫          │
               │   ≥90% 自动 / 60-90% 询问       │
               └──────────┬────────────────────┘
                          │ 发现新模式 →
                          │ 高置信自动建规则 /
                          │ 低置信建议人确认
                          ▼
    ┌──────────────────────────────────────────┐
    │    Tier 1 · 参数自调                       │
    │   每次交互都在学 · 全自动                    │
    │   路线workability / 时段偏好 / 时长 / buffer │
    └──────────────────────────────────────────┘
```

**本质变化：** 旧架构是 "agent 不进化，公司进化；公司进化了，agent 才进化"。新架构是 **"agent 在 Tier 1/2 内自主进化，Tier 3 才需要公司（人）介入"**。Agent 不等你——它自己变聪明，只在高风险或新种类的决策上敲门。

## 四、比架构更重要的：确认信号驱动飞轮

逻辑可以被抄，**积累的偏好数据 + agent 自我改进的速度**抄不走。

### 确认信号（全结构化，零 LLM 成本）
每次交互的确认/拒绝/修改/无视都是训练信号：

| 信号 | 喂给哪个 Tier | 含义 |
|------|:--:|------|
| 选了第 N 个建议 | Tier 1 | 排序偏好 |
| 拒绝了全部建议 | Tier 2 | 候选池/门控可能有问题 |
| 手动改了时长 | Tier 1 | 时长估算不准 |
| 手动加了 buffer | Tier 2 | buffer 规则可能缺失新维度 |
| 直接无视 | Tier 2 | 触发时机/优先级不对 |
| 改后自己又改回来 | Tier 1 | 撤销该条自动调整 |

### 埋点 schema（从第一天开始）

每次 agent 给出建议时记一条事件：
- **建议上下文**：通勤段（路线/模态/时长）、候选池来源、Top3 及其排序依据
- **用户反应**：接受 / 拒绝 / 选了第几个 / 改了什么字段 / 直接无视 / 反应耗时
- **执行结果**：排进去的事后来发生了吗 / 被改期 / 被删
- **agent 状态**：当时的 autonomy 滑块位置、已积累的置信度数据量

---

## 五、一句话收尾

> **旧：** 流程你写死（对），参数让数据调（真学习），新反馈靠"记日志 → 人+LLM 分析 → 上线"的元循环。
>
> **新：** 流程你写死（对），参数让数据调（Tier 1 · 全自动），新模式 agent 自己发现并决定（Tier 2 · 置信度门控），新维度靠"人+LLM 离线分析 → 上线 → 归入 Tier 1/2"（Tier 3 · 人拍板）。
>
> **Agent 不再等人——它自己越用越聪明，只在新种类的决策上敲门。产品真正的护城河不是那套逻辑，是确认信号飞轮 + 三层自我改进 + agent 自主进化的速度。**

## 六、强化版自我进化机制（重要修正）

三层架构是对的，但如果实现得太朴素，会在早期信任最脆弱的时候出问题。下面是必须补上的机制。

### 1. 小样本置信度陷阱

不要用裸比例判断置信度。`7/7` 看起来是 100%，但 `1/1` 也是 100%。早期数据少时，如果 agent 过早自动改规则，会让用户觉得它乱学。

**改法：用 Beta 后验下界，而不是裸百分比。**

- 每个偏好维护 `Beta(alpha, beta)`。
- 接受：`alpha += 1`；拒绝：`beta += 1`。
- 置信度看后验的保守下界，而不是均值。
- 效果：少量样本只会让 agent 继续观察；数据足够一致时才敢自动应用。

今天 MVP 不需要完整统计库，先用统一偏好 schema + 简化版置信度即可，避免裸 `+1/-1`。

### 2. 人会变，所以偏好必须会遗忘

一个学过就忘不掉的 agent，不是真正学习。路线会变、工作习惯会变、用户某段时间的偏好也会变。

**改法：时间衰减。**

每条信号带权重：

```text
w = 0.5 ^ (age / halflife)
```

半衰期可以先设 30-60 天。新行为会自然压过旧行为。

### 3. 接受不等于真实偏好

用户接受一个建议，可能只是因为它排第一、当时赶时间、或者没有更好的选项。只学接受率会导致 agent 过早收敛，只推荐自己已经相信的东西。

**改法：Beta 后验 + Thompson Sampling。**

排序时从每个候选的后验分布中采样：

- 高置信偏好更常排前面（利用）。
- 不确定但可能好的选项偶尔被试探（探索）。
- 同一个机制同时解决小样本、探索、位置偏差和不确定性。

这不是今天 MVP 必做，但应该作为 Tier 1 排序引擎的升级方向。

### 4. 学习最终结果，而不只是点击瞬间

真正重要的不是用户点了接受，而是安排是否最终发生。

**延迟奖励：**

- 接受并最终发生：强正反馈。
- 接受后取消 / 改期 / 用户手动撤销：强负反馈。
- 只接受但没有结果信号：弱正反馈。

这样 agent 学到的是"真实有效的安排"，不是"看起来用户点了的安排"。

### 5. 信号分层：无视不能当强负反馈

`无视` 很含糊：可能没看到、在忙、通知不合适，也可能是真的不想要。不能把它当明确拒绝。

信号强度从高到低：

```text
手动纠正 > 明确拒绝 > 明确接受 > 最终发生/取消 > 隐式无视
```

并且要区分：

- 用户看见后没动：弱负反馈。
- 根本没展示到用户眼前：不计入负反馈。

### 6. 偏好库：Agent 眼中的你

最好的进化机制不是黑箱学习，而是让用户能看见和纠正 agent 学到了什么。

未来需要一个 **"Agent 眼中的你"** 页面，展示偏好库：

```text
火车上可开视频 · 置信 88% · 基于最近 9/10 次 · 可撤回
午饭 12:00-13:00 不排会 · 置信 91% · 基于最近 10 次拒绝 · 可编辑
和 UZH 相关的人优先约大学附近 · 待确认 · 来自离线分析
```

每条偏好使用统一 schema：

```text
{
  dimension: "route_workability" | "time_preference" | "person_duration" | "buffer" | ...,
  key: string,
  value: unknown,
  alpha: number,
  beta: number,
  confidence: number,
  sampleCount: number,
  lastUpdated: timestamp,
  source: "explicit" | "behavior" | "inferred" | "offline_analysis",
  status: "active" | "suggested" | "trial" | "reverted"
}
```

人工纠正是最高质量训练信号，权重应高于隐式行为。

### 7. 安全信封：学习不能覆盖不变量

有些规则永远不能被学习覆盖：

- 不重复预订。
- 不在睡眠/明确不可用时间排事。
- 不在未授权时发消息。
- 不把私人/敏感事项暴露给对方。
- 不把低置信规则变成硬约束。

Tier 2 自动发现的新规则应先进入 `trial` 状态：先作为软提示运行，被推翻 N 次自动回滚。

### 8. 冷启动：用人群先验，然后快速个性化

新用户没有数据时，用人群先验初始化 Beta：

- 火车通常可用。
- 开车通常不可用。
- 公交/电车偏音频、轻任务。
- 飞机偏离线任务。

这些只是先验，不是最终判断。用户行为进来后逐渐覆盖。

### 对下一步的直接影响（更新）
- **今天 MVP**：Loop1 升级为 Tier 1 基础版。使用统一偏好 schema + 简化 Beta 置信度，不再裸 `weight+1/-1`。
- **今天 MVP**：埋点至少记录明确接受、明确拒绝、选择第几个、是否涉他人、是否生成/发送草稿。
- **本周**：补时间衰减、弱/强反馈权重、延迟奖励字段。
- **下次迭代**：Tier 2 模式发现引擎（统计扫描 + 置信度门控 + trial 状态）。
- **远期**：`Agent 眼中的你` 偏好库页面 + Tier 3 离线报告。
